## 1. บทนำ

[ชื่อผู้ควบคุมข้อมูล/ชื่อบริษัท] ("เรา") เป็นผู้พัฒนาและให้บริการแพลตฟอร์ม BP Monitor ซึ่งเป็นเครื่องมือช่วยบันทึกค่าความดันโลหิตและสนับสนุนการทำงานของอาสาสมัครสาธารณสุข ("บริการ") ผ่านทางเว็บแอปพลิเคชัน, PWA (Progressive Web App), Telegram Bot และ Telegram Mini App

เราตระหนักถึงความสำคัญของข้อมูลส่วนบุคคลและข้อมูลสุขภาพของท่าน นโยบายฉบับนี้จัดทำขึ้นเพื่อแจ้งให้ท่านทราบว่าเราเก็บรวบรวม ใช้ และเปิดเผยข้อมูลส่วนบุคคลของท่านอย่างไร ตามพระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 ("PDPA")

**ข้อมูลสุขภาพเป็น "ข้อมูลอ่อนไหว" (Sensitive Personal Data) ตามมาตรา 26 แห่ง PDPA** เราจะประมวลผลข้อมูลประเภทนี้เฉพาะเมื่อได้รับความยินยอมโดยชัดแจ้ง (Explicit Consent) จากท่านเท่านั้น

## 3. ข้อมูลที่เราเก็บรวบรวม

### 3.1 ข้อมูลส่วนบุคคลทั่วไป

#### ชาวบ้าน/ผู้ป่วย (ทั้ง self-managed และ proxy-managed)
- **ข้อมูลบัญชี**: ชื่อ-นามสกุล, อีเมล (optional สำหรับ proxy), หมายเลขโทรศัพท์ (optional สำหรับ proxy)
- **ข้อมูลระบุตัวตน**: เลขบัตรประชาชน (optional), วันเกิด, เพศ
- **ข้อมูลที่อยู่**: ที่อยู่ (สำหรับ proxy-managed เพื่อให้ อสม. ไปเยี่ยม)
- **ข้อมูลบัญชี Telegram**: Telegram User ID (เข้ารหัส) — เฉพาะผู้ใช้ที่เชื่อม Telegram

#### อสม. (NEW)
- ชื่อ-นามสกุล, หมายเลขโทรศัพท์, เลขบัตรประชาชน (optional)
- รพ.สต. ต้นสังกัด
- บทบาทในองค์กร
- Telegram User ID (หลัง pair สำเร็จ)

#### แพทย์
- ชื่อ-นามสกุล, อีเมล, หมายเลขโทรศัพท์
- **เลขใบอนุญาตประกอบวิชาชีพเวชกรรม** (ส่งตรวจกับแพทยสภา)

#### Admin ขององค์กร (NEW)
- ชื่อ-นามสกุล, อีเมล, หมายเลขโทรศัพท์
- บทบาทในองค์กร (ผู้มีอำนาจผูกพันองค์กร)
- IP และ user agent ตอนยอมรับ Org ToS (audit trail)

#### องค์กร (NEW)
- ชื่อองค์กร
- ประเภท (รพ.สต., คลินิก, ฯลฯ)
- รหัส HCODE (สำหรับ รพ.สต.)
- ที่อยู่ (จังหวัด, อำเภอ, ตำบล)
- ข้อมูลติดต่อ (เบอร์, email)

#### ทั้งหมด (transversal)
- **รหัสผ่าน** (เก็บในรูปแบบ Bcrypt hash — ไม่เก็บ plain text)
- **ข้อมูลการชำระเงิน** (เฉพาะ Premium): หมายเลขอ้างอิงการโอน, จำนวนเงิน, ธนาคาร, ชื่อผู้โอน, วันเวลา, ภาพสลิป

### 3.2 ข้อมูลสุขภาพ (ข้อมูลอ่อนไหว)

- **ค่าความดันโลหิต**: Systolic, Diastolic, ชีพจร (Pulse)
- **ข้อมูลร่างกาย**: กรุ๊ปเลือด, ส่วนสูง, น้ำหนัก
- **ข้อมูลการวัด**: วันเวลาที่วัด, บริบท (บ้าน/คลินิก/เยี่ยมบ้าน/ชุมชน), ใครวัด (ตัวคนไข้/อสม./เจ้าหน้าที่)
- **Location (optional)**: GPS coordinates ตอนวัด (เฉพาะเมื่อ อสม. ลงพื้นที่)
- **Notes**: บันทึกเพิ่มเติม
- **OCR metadata**: confidence score, raw Gemini output (JSON text)

### 3.3 ข้อมูลที่เกี่ยวข้องกับ Consent (NEW — v2)

- **Digital signature** (base64 PNG, encrypted) — ลายเซ็น e-signature ของชาวบ้าน
- **GPS coordinates** ตอนเก็บ consent
- **Timestamp** ตอนให้ consent
- **Witness user ID** (อสม. ที่เป็นพยาน)
- **Granular scopes** ที่ให้ consent (asm_collect, rpsst_view, doctor_view, etc.)
- **Paper filing metadata**: สถานที่เก็บกระดาษ, วันที่ filed (**ไม่ scan รูปกระดาษ**)

### 3.4 ภาพถ่าย (ชั่วคราว)

- **OCR Batch image** (ใบรายชื่อ + เครื่องวัด): เก็บ **ชั่วคราวสูงสุด 7 วัน** เฉพาะเมื่อ OCR confidence ต่ำและต้อง review. ลบอัตโนมัติหลัง admin review
- **OCR Single image** (จอเครื่องวัด 1 คน): **ไม่เก็บ** — process แล้วทิ้งทันที

### 3.5 ข้อมูลทางเทคนิค

- **ข้อมูลการเข้าใช้งาน**: หมายเลข IP, ข้อมูลอุปกรณ์ (User Agent), Session Token
- **บันทึกการใช้งาน**: เวลาเข้าใช้, จำนวนครั้งที่ลงชื่อเข้าใช้ผิดพลาด, เวลาที่ใช้งานล่าสุด
- **Request IDs** (สำหรับ correlation ใน audit log)

### 3.6 ข้อมูลความสัมพันธ์

- **การผูกแพทย์-ผู้ป่วย** (patient-doctor access requests)
- **Care assignments** (อสม. ดูแลใครบ้าง — NEW)
- **Organization memberships** (ใครเป็นสมาชิกของ รพ.สต. ไหน — NEW)

### 3.7 Audit Log (NEW — v2)

เราบันทึก **ทุก action ที่เข้าถึงข้อมูลข้ามผู้ใช้** เพื่อให้สามารถตรวจสอบย้อนหลังได้:
- ใครเข้าถึงข้อมูลของใคร, เมื่อไหร่, ผ่าน endpoint อะไร
- IP, user agent, request ID
- ผลลัพธ์ (สำเร็จ/ปฏิเสธ)

รายละเอียดใน Section 7.2

### 3.8 ข้อมูลที่เราไม่เก็บ (by design)

- ❌ **รูปจอเครื่องวัด single OCR** — ลบทันทีหลัง process
- ❌ **รูปกระดาษ consent form** — เก็บ physical ที่ รพ.สต. ไม่ scan เข้าระบบ (data minimization)
- ❌ **รูปใบหน้าชาวบ้าน** — ไม่จำเป็น เพราะมี e-signature + witness + GPS
- ❌ **เนื้อหาการสนทนาใน Telegram** นอกเหนือจากที่ bot ต้อง process
- ❌ **Behavioral analytics** — ไม่ track การใช้งานเพื่อ marketing
- ❌ **Third-party cookies** — ไม่ใช้

### 4.1 Granular Consent Scopes (NEW — v2)

ระบบรองรับ consent แบบ granular — ชาวบ้านเลือกยินยอมเรื่องใดได้บ้าง:

| Scope | ความหมาย | จำเป็น? |
|-------|---------|--------|
| `asm_collect` | ให้ อสม. เก็บข้อมูลความดันของตน | **Core** (ต้องมี) |
| `rpsst_view` | ให้ รพ.สต. ดูข้อมูลเพื่อการดูแล | **Core** (ต้องมี) |
| `doctor_view` | ให้แพทย์ที่อนุมัติดูข้อมูล | Optional |
| `research_anonymized` | ให้ใช้ข้อมูลเพื่อ research (ไม่ระบุตัวตน) | Optional |
| `data_export_to_smart_osm` | ส่งข้อมูลไป Smart อสม. (กระทรวงสาธารณสุข) | Optional |

ท่านถอน consent scope ใดก็ได้ทุกเมื่อ — ระบบจะ enforce ตามทันที (block การเข้าถึงที่อาศัย scope นั้น)

### 5.2 บุคคลอื่นที่อาจเข้าถึงข้อมูล

- **แพทย์ที่ท่านอนุมัติ** (self-managed flow): เข้าถึงข้อมูลสุขภาพของท่านเฉพาะเมื่อท่านอนุมัติคำขอเชื่อมต่อแล้ว ท่านสามารถเพิกถอนการเข้าถึงได้ตลอดเวลา
- **อสม. ที่ได้รับมอบหมาย** (NEW — v2): เข้าถึงข้อมูลชาวบ้านที่ตัวเองได้รับการผูกผ่าน care assignment เท่านั้น เมื่อชาวบ้านได้ให้ consent scope "asm_collect"
- **เจ้าหน้าที่ รพ.สต. ที่เป็น admin** (NEW — v2): เข้าถึงข้อมูลชาวบ้านทุกคนในความรับผิดชอบของ รพ.สต. นั้น เมื่อได้ให้ consent scope "rpsst_view"
- **หน่วยงานของรัฐ**: เมื่อมีคำสั่งที่ชอบด้วยกฎหมายเท่านั้น

**ข้อจำกัด (Organization Isolation):**
- อสม. และ admin เห็นเฉพาะข้อมูลของ **องค์กรของตัวเอง** เท่านั้น
- **ทุก access ถูก log ใน audit trail** (ดู Section 7.2)
- ข้อมูลข้ามองค์กร ต้องผ่าน data portability process

## 6. การโอนข้อมูลไปต่างประเทศ

เนื่องจากผู้ให้บริการภายนอกที่เราใช้บางรายมีที่ตั้งอยู่นอกประเทศไทย ข้อมูลส่วนบุคคลของท่าน **อาจถูกโอนไปยังต่างประเทศ** ได้แก่ สหรัฐอเมริกา สหรัฐอาหรับเอมิเรตส์ และสหราชอาณาจักร

เราดำเนินมาตรการเพื่อให้การโอนข้อมูลเป็นไปตามมาตรา 28 แห่ง PDPA ดังนี้:

- คัดเลือกผู้ให้บริการที่มีมาตรฐานการคุ้มครองข้อมูลส่วนบุคคลที่เพียงพอ
- ผู้ให้บริการทั้งหมดมีนโยบายความเป็นส่วนตัวและมาตรฐานความปลอดภัยระดับสากล (เช่น SOC 2, ISO 27001)
- ข้อมูลที่จัดเก็บในฐานข้อมูลถูก **เข้ารหัส (Encrypted)** ก่อนส่งออกจากเซิร์ฟเวอร์ของเรา ดังนั้นผู้ให้บริการฐานข้อมูลจะเห็นเฉพาะข้อมูลที่เข้ารหัสแล้วเท่านั้น ไม่สามารถอ่านข้อมูลดิบได้
- การสื่อสารทั้งหมดใช้โปรโตคอล HTTPS/TLS เพื่อเข้ารหัสข้อมูลระหว่างส่ง

**แผนในอนาคต:** เรามีแผน migrate ระบบไปยัง cloud ในประเทศไทยเพื่อ data residency ในช่วง Phase 2 ของโครงการ

## 7. มาตรการรักษาความปลอดภัย

### 7.1 การเข้ารหัสข้อมูล (Encryption)

- **ข้อมูลส่วนบุคคลที่ระบุตัวตนได้** (ชื่อ, อีเมล, เบอร์โทร, เลขบัตรประชาชน, เลขใบอนุญาตแพทย์, วันเกิด, Telegram ID, digital signature) ถูก **เข้ารหัสด้วย Fernet Symmetric Encryption (AES-128-CBC)** ก่อนจัดเก็บในฐานข้อมูล
- **ฟิลด์ที่ต้องค้นหา** ถูก **แฮช (SHA-256 Hash)** แยกต่างหากเพื่อใช้ในการค้นหาโดยไม่ต้องถอดรหัสข้อมูลทั้งหมด
- **รหัสผ่าน** เก็บด้วย **Bcrypt Hash** เท่านั้น ไม่เก็บ plain text
- **Pairing codes** เก็บเฉพาะ SHA-256 hash ไม่เก็บ code ตรง ๆ

### 7.2 การควบคุมการเข้าถึง (RBAC — NEW v2)

- **Role-Based Access Control**: 6 บทบาท (superadmin, rpsst_admin, doctor, asm, patient_self, patient_proxy)
- **Tenant isolation**: อสม./admin เห็นเฉพาะข้อมูลในองค์กรของตัวเอง
- **Care assignment enforcement**: อสม. เห็นเฉพาะชาวบ้านที่ถูกผูก
- **Consent scope enforcement**: ทุก API check ว่า consent scope active ไหม ก่อนให้ access
- **JWT Token** + กำหนดอายุการใช้งาน (ที่แตกต่างกันตามบทบาท)
- **OTP 2FA** (SMS / Email / Telegram)
- **Account lockout** หลังล็อกอินผิดหลายครั้ง
- **Rate limiting** ป้องกัน brute force

### 7.3 Audit Log (NEW — v2)

เรา log ทุก **cross-user access** และ security-sensitive actions:

**Events ที่ log:**
- Login success/failure, password changes
- User create/suspend/deactivate
- Care assignment changes
- Patient data views (by non-owner)
- BP reading create/update/delete
- Consent grant/withdraw
- Data exports
- OCR batch processing

**Fields:**
- Actor (ใครทำ), Target (ทำอะไรกับใคร)
- IP, User Agent, Request ID
- Timestamp, success/failure, metadata

**Retention:** 2 ปี hot (ใน DB) + 5 ปี cold (archive)

### 7.4 ความปลอดภัยในการส่งข้อมูล

- การสื่อสารทั้งหมดระหว่างอุปกรณ์ของท่านกับเซิร์ฟเวอร์ใช้ HTTPS (TLS 1.2 ขึ้นไป)
- API ทั้งหมดต้องมี JWT token ประกอบการเรียกใช้

### 7.5 ความปลอดภัยของภาพถ่าย

- **OCR Single image**: ส่ง Gemini API → ลบจากหน่วยความจำทันทีหลัง response กลับ **ไม่มีการเก็บถาวรเลย**
- **OCR Batch image**: เก็บเฉพาะเมื่อ confidence ต่ำ + ต้อง review โดย admin. **อายุสูงสุด 7 วัน**. ลบอัตโนมัติหลัง admin ดำเนินการ review
- ภาพที่เก็บชั่วคราวถูก encrypt ด้วย Fernet
- เราไม่ใช้ภาพของท่านเพื่อฝึกโมเดล AI ของเรา

### 7.6 Tamper detection

- Digital signatures มี SHA-256 hash เก็บแยก (detect modification)
- Audit logs เป็น append-only (ไม่สามารถ UPDATE/DELETE)

## 8. ระยะเวลาในการเก็บรักษาข้อมูล

(อัปเดตเพื่อสอดคล้องมาตรฐานเวชระเบียน — 10 ปี)

| ประเภทข้อมูล | ระยะเวลา | เหตุผล |
|-------------|---------|--------|
| ข้อมูลบัญชี (self_managed) | ขณะ active + 90 วันหลังลบบัญชี | สัญญา + กฎหมายที่เกี่ยวข้อง |
| ข้อมูลบัญชี (proxy_managed) — ชาวบ้าน | ขณะ active + 90 วันหลังถอน consent | PDPA + เวชระเบียน |
| ข้อมูลบัญชี อสม. | ขณะยังเป็น org member + 1 ปี | Audit trail |
| ข้อมูลสุขภาพ (ค่าความดัน) | **10 ปี** หรือจน withdraw consent + erase | **มาตรฐานเวชระเบียน** (กฎกระทรวง พ.ศ. 2562) |
| Consent records | **10 ปีหลัง withdrawal** | PDPA obligation (legal defensibility) |
| Digital signature ใน consent | 10 ปีหลัง withdrawal | Legal evidence |
| Care assignments | ขณะ active + 10 ปี | Audit + clinical continuity |
| ภาพ OCR Batch (review queue) | **สูงสุด 7 วัน** | Data minimization |
| ภาพ OCR Single | ไม่เก็บ (ลบทันที) | Data minimization |
| รหัส OTP | หมดอายุอัตโนมัติภายใน 5 นาที | ความจำเป็นทางเทคนิค |
| Pairing codes | 15 นาที | ความจำเป็นทางเทคนิค |
| ข้อมูล Session / JWT | ตามอายุ token | ความปลอดภัย |
| ข้อมูลการชำระเงิน | 5 ปีนับจากวันทำรายการ | ภาษีและกฎหมายการเงิน |
| **Audit logs** | **2 ปี hot + 5 ปี cold** | PDPA + security forensics |
| บันทึกความปลอดภัย (security logs) | 90 วัน | ตรวจสอบเหตุการณ์ |

เมื่อครบกำหนดหรือเมื่อท่านใช้สิทธิลบข้อมูล เราจะลบหรือทำให้ข้อมูลไม่สามารถระบุตัวตนได้ (anonymize)

## 9. สิทธิของท่านตาม PDPA

ท่านมีสิทธิตามกฎหมายดังต่อไปนี้:

**9.1 สิทธิในการเข้าถึง (Right of Access)** — ขอสำเนาข้อมูลส่วนบุคคลของท่านที่เราเก็บรวบรวมไว้

**9.2 สิทธิในการแก้ไข (Right to Rectification)** — ขอให้แก้ไขข้อมูลที่ไม่ถูกต้อง

**9.3 สิทธิในการลบ (Right to Erasure)** — ขอให้ลบข้อมูลส่วนบุคคลของท่าน (เว้นแต่มีหน้าที่ตามกฎหมายที่ต้องเก็บ)

**9.4 สิทธิในการระงับการประมวลผล (Right to Restriction)** — ขอให้ระงับการใช้ข้อมูลชั่วคราว

**9.5 สิทธิในการโอนย้ายข้อมูล (Right to Data Portability)** — ขอรับข้อมูลของท่านในรูปแบบที่อ่านได้ด้วยเครื่อง (CSV, JSON) ท่านสามารถใช้ฟีเจอร์ "ส่งออกข้อมูล" ในแอปพลิเคชันได้

**9.6 สิทธิในการคัดค้าน (Right to Object)** — คัดค้านการประมวลผลข้อมูลที่อาศัยฐานประโยชน์โดยชอบด้วยกฎหมาย

**9.7 สิทธิในการถอนความยินยอม (Right to Withdraw Consent)** — ท่านสามารถถอนความยินยอมได้ตลอดเวลา (ทั้ง scope เดียวหรือทั้งหมด)

### 9.8 สิทธิของ Proxy-managed Patients (NEW — v2)

หากท่านเป็นชาวบ้านที่ข้อมูลถูกเก็บโดย อสม. (proxy-managed) ท่านมีสิทธิเต็มตาม PDPA เหมือนผู้ใช้ทั่วไป และเพิ่มช่องทางพิเศษ:

- **ขอดูข้อมูล**: ติดต่อ รพ.สต. หรือ อสม. ที่ดูแล
- **ขอแก้ไข**: แจ้ง อสม. หรือ admin รพ.สต.
- **ขอลบ**: ติดต่อ รพ.สต. admin — admin จะดำเนินการในระบบ
- **ถอน consent**: ทำได้ตลอดเวลา — verbal, in-person, หรือจดหมาย
- **โอนข้อมูล**: ขอ export เป็น PDF หรือ CSV จาก admin

**ช่องทางพิเศษ:** หากท่านไม่สามารถเข้าถึง อสม./รพ.สต. ได้ (เช่น ย้ายออกจากพื้นที่) ติดต่อเราโดยตรงที่ [privacy@yourdomain.com]

**เวลาตอบสนอง:** เราจะดำเนินการภายใน **30 วัน** นับแต่วันที่ได้รับคำขอพร้อมข้อมูลยืนยันตัวตนของท่าน (อาจผ่าน รพ.สต.)

## 10. การประมวลผลข้อมูลในบริบทเฉพาะ

### 10.1 Telegram Bot และ Mini App

เมื่อท่านใช้บริการผ่าน Telegram:
- เราเก็บ Telegram User ID ของท่านเพื่อเชื่อมต่อกับบัญชีในระบบ
- ข้อความที่ท่านส่งถึง Bot (เช่น ค่าความดัน) จะถูกประมวลผลผ่านเซิร์ฟเวอร์ของ Telegram ก่อนส่งมาถึงระบบของเรา
- เราไม่สามารถควบคุมการเก็บข้อมูลโดย Telegram ได้ — ตรวจสอบนโยบายของ Telegram ที่ https://telegram.org/privacy

### 10.2 การอ่านค่าจากภาพ (OCR)

เมื่อท่าน/อสม. อัปโหลดภาพเครื่องวัดหรือใบรายชื่อ:
- ภาพจะถูกส่งไปยัง Google Gemini API เพื่อสกัดค่า
- Google ประมวลผลภาพตามนโยบายของ Google Cloud — https://cloud.google.com/terms/data-processing-addendum
- ภาพจะถูกลบจากเซิร์ฟเวอร์ของเราทันทีหลังประมวลผล (OCR Single) หรือเก็บชั่วคราว 7 วัน (OCR Batch — review queue)

### 10.3 การตรวจสอบคุณสมบัติแพทย์

สำหรับผู้ลงทะเบียนเป็นแพทย์:
- เราส่งชื่อและเลขใบอนุญาตไปตรวจสอบกับแพทยสภาแห่งประเทศไทย (TMC)
- ข้อมูลผู้ป่วยจะไม่ถูกส่งไปในกระบวนการนี้

### 10.4 การชำระเงิน

- ภาพสลิปโอนเงินจะถูกส่งไปตรวจสอบกับ SlipOK API
- เราไม่เก็บข้อมูลบัตรเครดิตหรือข้อมูลบัญชีธนาคารของท่าน

### 10.5 Care Assignment (NEW — v2)

เมื่อ admin ผูก อสม. กับชาวบ้าน:
- สร้าง record ใน `care_assignments` พร้อม effective date
- Audit log บันทึกผู้สร้าง assignment
- เมื่อ end assignment: บันทึกเหตุผล (transfer, ชาวบ้านย้าย, อสม. ลาออก)
- ข้อมูลประวัติ assignment เก็บไว้ 10 ปี (audit)

### 10.6 Consent Capture (NEW — v2)

เมื่อ อสม. เก็บ consent:
- ชาวบ้านเซ็นกระดาษ → อสม. เก็บกระดาษไปให้ admin
- อสม. กรอกข้อมูลใน PWA → ชาวบ้านแตะนิ้วเซ็น e-signature บนจอ
- ระบบเก็บ: encrypted signature PNG + GPS + timestamp + witness (อสม.)
- กระดาษเก็บใน ตู้ล็อก ที่ รพ.สต. — **ไม่ scan เข้าระบบ**

## 14. Data Breach Notification

หากเกิดเหตุการละเมิดข้อมูลส่วนบุคคลที่มีนัยสำคัญ เราจะ:

1. **แจ้งสำนักงาน สคส. ภายใน 72 ชั่วโมง** (ตาม ม.37 PDPA)

3. **สำหรับ joint controller mode:** แจ้ง รพ.สต. ภายใน 24 ชั่วโมง
4. **ประเมินขอบเขต, containment, remediation** ตาม internal breach response runbook
